# 建立 SDD 开发 Harness

## 背景

仓库原本已有应用代码和 Docker 部署，但缺少明确的变更生命周期、自动化测试、
lint、CI 和一键质量门。后续开发者和智能体工作时没有可执行的完成标准。

## 目标

- 将需求、设计、任务和验证证据作为仓库中的一等数据管理。
- 让本地开发、智能体和 CI 使用同一套确定性的质量门。
- 记录稳定的应用架构边界和后续变更流程。

## 非目标

- 不重新设计产品，也不替换现有 Go/React 架构。
- 不增加依赖数据库的集成测试或生产部署自动化。

## 需求

- R1：提供命令创建完整的变更规格目录。
- R2：提供命令校验规格结构和 `verified` 状态的不变量。
- R3：通过一个命令完成规格、静态质量、测试和构建检查。
- R4：CI 必须执行与本地相同的完整验证命令。
- R5：仓库文档必须说明开发者和智能体何时以及如何使用 SDD。

## 验收标准

- [x] `npm run spec:new -- <slug> "<标题>"` 能创建全部必要的 SDD 文件。
- [x] `npm run spec:check` 能拒绝结构错误或证据不完整的已验证变更。
- [x] `npm run verify` 无需启动基础设施即可运行规格、Go 和前端质量门。
- [x] CI 能安装依赖并调用同一个 `npm run verify` 命令。
- [x] 仓库中可以直接找到架构、生命周期和智能体协作规则。
