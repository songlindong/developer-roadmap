# 建立 SDD 开发 Harness——实现计划

## 设计方案

使用 Markdown 记录适合人阅读的设计推理，使用 `change.json` 记录机器可读的
生命周期状态。无第三方依赖的 Node 执行器读取 `harness/config.json` 中的声明式
命令，校验规格并按顺序执行质量门。根目录 npm scripts 提供简短、跨平台的统一入口。
GitHub Actions 调用同一入口，避免重复维护验证逻辑。

## 影响范围

新增仓库流程文件、前端 lint/测试脚本、带测试的前端纯函数、后端单元测试和 CI。
不改变 HTTP 契约、数据结构、运行时配置或已部署的产品行为。

## 测试策略

- 使用本变更规格测试规格校验，并创建一条临时变更测试脚手架命令。
- 运行 Go 单元测试、vet、格式检查和构建。
- 运行前端单元测试、lint 和 Vite 生产构建。
- 最终运行 `npm run verify`，端到端验证完整 Harness。

## 发布与回滚

无需运行时发布。后续有实际意义的变更开始采用此流程。需要回滚时可以删除流程文件和
package scripts，应用运行时行为不会受到影响。
